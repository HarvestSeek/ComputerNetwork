    3.Connection-Oriented Transport： TCP

    1. TCP 报文段结构


    TCP 的头部一般为 20 字节。
    ① 源端口号/目的端口号
     同 UDP 一样，用于多路复用/多路分用。

    ② 序列号、确认号 (32 bit)
     TCP 把数据视为有序的字节流，序列号基于字节顺序而不是报文段的序号。当接收方收到报文段之后，接收方填进报文段的序列号是它期望从发送方收到的下一字节的序号。
     TCP 是提供累积确认的协议，对于接收方来说，如果收到失序分组，仍会返回最后收到的有序字节号+1的确认号。TCP RFC 没有明确规定对失序报文段的处理，它的具体实现取决于 TCP 实现者，实践中接收方保留失序字节、等待缺少的字节对于网络带宽更有效。
     TCP 双方的初始序列号一般为随机选取，这样可以减少两台主机之间先前已终止的连接发送的报文段被误认为是新连接的报文段的可能性。

    ③ 首部长度 (4 bit)
     指示了以 32 比特的字为单位的 TCP 首部长度。由于 option 字段的原因，TCP 首部长度可变。option 字段通常为空，因此 TCP 首部字段长度一般为 20 字节。

    ④ 标志字段 (6 bit)
      ACK: 用于指示确认号字段的值有效。
    RST、SYN、FIN: 用于连接的建立和拆除。

    ⑤ 接收窗口字段 (16 bit)
     用于流量控制。

    ⑥ 选项字段
     用于发送方和接收方协商最大报文段长度，或在高速网络环境下作窗口调节因子使用。

    2. 往返时间估计 (Round-trip Time Estimation)

     TCP 仍使用超时/重传机制处理数据丢失问题，并维护一个动态的样本往返时间 $SampleRTT$。大多数 TCP 仅在某个时刻对于一个已发送但未确认的报文段做一次 $SampleRTT$ 的测量，而忽略被重传的报文段。TCP 还维护一个样本 RTT 的平均值，每当获得一个新的样本值时更新均值： $$EstimatedRTT=(1-\alpha)\cdot EstimatedRTT+\alpha\cdot SampleRTT$$ 其中 $\alpha$ 的推荐值为 0.125。$EstimatedRTT$ 是 $SampleRTT$ 的一个加权平均值，在统计学上，这种平均被称为指数加权移动平均 (Exponential Weighted Moving Average, EWMA)。
     TCP 还维护一个 RTT 偏差值，用于估算 $SampleRTT$ 偏离 $EstimatedRTT$ 的程度： $$DevRTT=(1-\beta)\cdot DevRTT+\beta\cdot |SampleRTT-EstimatedRTT|$$ $\beta$ 的推荐值为 0.25。如果 $SampleRTT$ 的波动很小，那么 $DevRTT$ 的值也很小，反之则很大。
     TCP 超时间隔应设为大于等于 $EstimatedRTT$，否则会造成不必要的重传；但不能大太多，否则会徒增时延。 $$TimeoutInterval=EstimatedRTT+4\cdot DevRTT$$ $TimeoutInterval$ 的初值推荐为 1 秒。一旦出现超时，它的值加倍；只要收到报文段并更新 $EstimatedRTT$，就使用上述公式再次计算 $TimeoutInterval$。

 * 发送方不受 TCP 流量控制和拥塞控制的限制

 * 单向数据传送
 */
NextSeqNum = Initial Sequence Number;
SendBase = Initial Sequence Number;
while (true) {
    switch (event)
        event: data received from application above
            create TCP Segment with sequence number NextSeqNum;
            if (定时器未运行) 
                start timer;
            pass segment to IP;
            NextSeqNum = NextSeqNum + length(data);
            break;
        event: timer timeout
            retransmit not-yet-acked segment with smallest sequence number;
            start timer;
            break;
        event: ACK received, with ACK field value of y
            if (y > SendBase) {
                SendBase = y;
                if (there are currently any not-yet-acked segment) {
                    start timer;
                }
            }
            break;
}
     TCP 采用单一定时器。注意到，如果收到的 ACK 使得 send_base 刷新，且还有已发送但未收到回应的报文段的话，要重启计时器。

    (2). 超时间隔加倍

     在大多数 TCP 中，每当超时事件发生，在重传相应报文段后，会设置 $TimeoutInterval×2$；直到另外两个响应时间其一发生，通过 $EstimatedRTT$ 和 $DevRTT$ 得到超时间隔的新值。
     这种机制提供了一种形式受限的拥塞控制，定时器过期很可能由网络拥塞引起。

    (3). 快速重传 (Fast Retransmit)

     当一个报文段丢失，如果超时周期过长，会增加端到端时延。TCP 发送方通过检测重复 ACK (duplicate ACK) 实现快速重传机制：如果 TCP 发送方接收到对于同一个报文段的 3 个 ACK，就认为该报文段已丢失，在定时器过期之前执行快速重传。

    4. TCP 连接管理

    (1). TCP 通信过程

     在一个进程可以向另一个进程发送数据之前，这两个进程必须相互握手：发送预备报文段、确保传输的某些参数。这种连接是一种逻辑连接，连接的状态仅保留在端系统的 TCP 程序中，而中间的网络元素不保留关于连接的任何信心。
     在建立连接后，进程之间就可以相互发送数据。发送方通过 Socket 得到上层的数据后，将其引导至发送缓存中。TCP 从缓存中取出的数据大小受限于最大报文段长度 (Maximum Segment Size, MSS)，MSS 通常根据最大链路层帧长度 (最大传输单元, Maximum Transmission Unit, MTU) 设置。TCP 为每个数据配上头部，形成报文段，下传给网络层。接收方将接收到的数据引导至接收缓存中。
     因此，TCP 连接的组成包括：一台主机上的缓存、变量、套接字以及另一台主机上的相同元素。

    (2). TCP 连接建立：三次握手

     以客户端主动向服务器发起连接为例。

    • ① 客户端向服务器发送一个 SYN 报文段，在这个报文段中，标志位 SYN 被置为 1 (ACK = 0)，并设置了一个随机的初始序列号 client_isn。
      • C: to SYN_SENT
    • ② 服务器收到后，首先为 TCP 连接分配缓存和变量，然后发送一个 SYNACK 报文段，其中标志位 SYN、ACK 被置为 1，表示允许连接；Ack = client_isn + 1，Seq = server_isn。
      • S: to SYN_RCVD
    • ③ 客户端收到 SYNACK 报文段后，为该连接分配缓存和变量。然后向服务器发送一个报文段，对服务器的允许连接进行确认，因为连接已经建立，SYN 被置为 0。这个报文段可以携带客户端要发送的数据。
      • C/S: to ESTABLISHED

     这个过程被称为三次握手 (three-way handshake)。在建立连接后，C/S 之间就可以相互发送数据了，但 SYN 被置为 0.

    (3). TCP 连接关闭：四次挥手

     以客户端主动发起关闭请求为例。

    • ① 客户端发送一个特殊的报文段，关闭客户端到服务器的数据传送，其中的 FIN 标志位置为 1。
      • C: to FIN_WAIT_1
    • ② 服务器收到后，发送一个确认报文段，客户端接收这个报文段。
      • S: to CLOSE_WAIT, C: to FIN_WAIT_2
    • ③ 服务器发送终止报文段，关闭服务器到客户端的数据传送，FIN 置为 1。
      • S: to LAST_ACK
    • ④ 客户端收到服务器的 FIN 后，发送一个确认报文段。
      • C: to TIME_WAIT, S: to CLOSED。

     被动方在收到主动方的关闭请求后，可能还有需要继续处理的报文段，因此，发送 ACK 和发送 FIN 要分开进行。
