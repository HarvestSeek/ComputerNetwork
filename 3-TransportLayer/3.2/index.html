    2.Principles of Reliable Data Transfer

    1. 可靠数据传输基本原理

    (1). 概述

     信道本身是不可靠的，通过数据传输协议的处理使传输变得可靠。可靠的定义为：不错、不丢、不乱。信道的不可靠性决定了可靠数据传输协议的复杂性。
     可靠数据传输协议的基本结构：


    应用层与传输层之间的调用是单向的：可靠数据传输底层实现交给传输层实现；传输层和网络层之间的调用时双向的：在不可靠信道上数据传输不仅需要单向的数据流动，还需要双向的控制信息流动。
     以下的讨论仅考虑单向 (unidirectional) 数据传输。

    (2). Rdt1.0: 可靠信道

     假设底层信道处于理想状态，完全可靠：不发生位错误、不丢弃分组、不乱序。
     发送方和接收方的状态机独立，不需要交互控制消息。

    (3). Rdt2.0: 产生位错误的信道

     假设信道只会产生位错误，底层信道可能翻转某些位。发送方可以使用 checksum 机制检测错误；如果要从错误中恢复，就要使用确认机制 (Acknowledgements, ACK)：接收方显式告诉发送方已正确接收；通过发送 NAK 告知发送方存在错误，发送方收到 NAK 后，重传分组。
     基于这种重传机制的 rdt 协议称为 ARQ(Automatic Repeat reQuest) 协议。ARQ 协议的实现需要三种功能：1.差错检测；2.接收方反馈；3.重传。


     当发送方处于等待 ACK/NAK 的状态时，不能从上层接收新数据，除非它确信接收方正确接收分组。这样的协议被称为停-等 (stop-and-wait) 协议。

    (4). Rdt2.1、2.2: 改进 Rdt2.0

     Rdt2.0 存在致命缺陷：无法处理 ACK/NAK 发生错误或者被破坏 (corrupted) 的情况。解决的办法是如果发送方收到损坏的 ACK/NAK 就让发送方重传，但简单的重传会导致重复分组 (duplicate packet) 问题：接收方可能收到相同的数据，因此 Rdt2.1 引入序列号 (sequence number) 机制：一旦重复，接收方丢弃重复分组。
     因为采用停-等协议，只需要两个序列号就足够，因此与 Rdt2.0 相比，状态数量翻倍。接收方需要判断是否重复：收到一个重复分组后，丢弃，并发送一个 ACK。



     从发送方的角度来说，如果收到一个 NAK，等同于接收方最后收到上一个成功发送的包。因此，Rdt2.2 在此基础上，使用 ACK + seq number 代替 NAK；从接收方来看，需要重传的两种情况行为相同，因此合并。


    (5). Rdt3.0

     信道既可能发生错误，也可能丢失分组。发送方发送的分组、接收方返回的 ACK 都可能会发生丢失，解决的方法是让发送方等待一个合理的时间，这个时间过后，如果没收到 ACK，则重传。如果是因为延迟问题，数据包并未真生丢失，可以通过序列号解决重复分组问题。


     对于发送方来说，分组丢失、ACK 丢失、ACK 损坏、ACK 的序列列号错误发生时都会导致超时，然后重传。如果定时器设置过短造成接收方在等待状态下收到损坏或不对应的 ACK 时会什么也不做，避免了不必要的重传。

     Rdt3.0 可以正确工作，但性能很差。停-等协议信道利用率：$\frac{L/R}{RTT+L/R}, RTT=2×d_{prop}$

    2. 滑动窗口协议

    (1). 概述

     针对停-等协议利用率低的原因，利用流水线机制，允许发送方在收到 ACK 之前连续发送多个分组，此时利用率变为 $\frac{n×L/R}{RTT+L/R}$。相应地，需要更大的序列号范围，同时发送方和接收方也需要更大的缓存空间。
     要想实现这种流水线机制，就需要滑动窗口协议 (sliding-window protocol)。窗口：允许使用的序列号范围，最多等待确认的消息数。滑动：随着协议运行，序列号在空间内向前滑动。

    (2). GBN 协议 (Go-Back-N)

     图示为发送方视角的序号范围。基序号 base 为最早已发送但未确认的序号，nextseqnum 为最小的未使用序号。



    发送方响应事件：
    ① 上层调用：发送方首先检查发送窗口是否已满，如果未满则生成分组并发送、并更新变量；否则，把数据返回给上层。（在实际实现时，使用缓存或同步机制)
    ② 超时：如果发生超时，发送方重传所有已发送但还未确认的分组；发送方仅使用一个定时器，它被用给最早发送但还未被确认的分组。在收到一个 ACK 后，如果仍有未被确认的分组，重启定时器；否则停止定时器。
    ③ 收到一个 ACK：GBN 协议对分组采用累积确认 (cumulative acknowledgment) 的方式。发送方更新 base 和定时器；对接收方来说，如果一个分组 n 被收到且按序，则接收方为 n 发送一个 ACK，表明它收到了 n 之前且包括 n 的所有分组，并向上层交付数据；在发送方的视角，可能没有收到某个 n-i 的 ACK，但只要收到 ACK n，send_base 就移动到 n + 1 的位置。所有其他情况下，接收方丢弃分组，并发送 ACK + 最近收到的按序分组序列号。

    (3) SR 协议 (Selective Repeat)

     GBN 协议仍存在性能问题：当窗口长度和时延带宽积很大，流水线中有很多分组时，单个分组的差错就会引起大量不必要的重传。

    发送方响应事件：
    ① 上层调用：同 GBN。
    ② 超时：每个分组都有自己的逻辑定时器。
    ③ 收到一个 ACK：如果收到的 ACK 在窗口内，则标记为已接收；如果收到的分组序号等于 send_base，窗口基序号移动到最小已发送未确认序号处；如果窗口发生移动，且有可发送序号，则发送这些分组。

    接收方响应事件：
    ① 在 [rcv_base, rcv_base + N - 1] 内的分组被接收：即收到的分组落在接收方的窗口范围内。首先返回一个 ACK；如果该分组以前未收到，缓存该分组；如果该分组等于基序号 rcv_base，则将连续的缓存分组交付上层，接收窗口移动。
    ② 在 [rcv_base - N, rcv_base - 1] 内的分组被接收：即接收方已确认接收并在窗口外的重复分组，返回一个 ACK。
    ③ 其他情况，忽略收到的分组。
