    2.Principles of Reliable Data Transfer

    1. 可靠数据传输基本原理

    (1). 可靠数据传输协议的基本结构

     信道本身是不可靠的，通过数据传输协议的处理使传输变得可靠。可靠的定义为：不错、不丢、不乱。信道的不可靠性决定了可靠数据传输协议的复杂性。
     可靠数据传输协议的基本结构：


    应用层与传输层之间的调用是单向的：可靠数据传输底层实现交给传输层实现；传输层和网络层之间的调用时双向的：在不可靠信道上数据传输不仅需要单向的数据流动，还需要双向的控制信息流动。
     以下的讨论仅考虑单向 (unidirectional) 数据传输，并始终假设分组在信道转发过程中不会乱序。

    (2). Rdt1.0: 可靠信道

     假设：底层信道处于理想状态，完全可靠。
     发送方和接收方的状态机独立，不需要交互控制消息。

    (3). Rdt2.0: 产生位错误的信道

     假设：信道只会产生位错误，底层信道可能翻转某些位。
     Rdt2.0 因此引入了检错机制和恢复机制。 发送方和接收方都通过计算 checksum 检错；如果发现错误，通过确认机制 (Acknowledgements) 从错误中恢复：

    • 接收方通过发送 ACK/NAK，显式通知发送方 正确接收/未正确接收
    • 发送方收到 NAK 后，重传分组
     基于这种重传机制的 rdt 协议称为 ARQ(Automatic Repeat reQuest) 协议。ARQ 协议的实现需要三种功能：1.差错检测；2.接收方反馈；3.重传。


     当发送方处于等待 ACK/NAK 的状态时，不能从上层接收新数据，除非它确认接收方正确接收分组。这样的协议被称为停-等 (stop-and-wait) 协议。

    (4). Rdt2.1、2.2: 改进 Rdt2.0

     假设：同 Rdt2.0。
     Rdt2.0 无法处理 ACK/NAK 发生错误或者被破坏的情况。解决的办法是如果发送方收到损坏的 ACK/NAK 就重传分组，但这会导致重复分组问题，即接收方无法知道收到的包是新包还是重传包。因此 Rdt2.1 又引入了序列号机制，由停等协议的特性以及不乱序的假设，有以下事实：

    • 接收方在正确收到 Packeti 后，下一个收到的包只可能是 Packeti 或 Packeti+1
     因此只需要两个序列号，Rdt2.1 的 FSM 如下所示。注意，接收方在收到序列号不匹配的包时，要返回一个 ACK。


     在 Rdt2.1 中，从发送方的视角看，“接收方没有正确接受 Packeti” 和 “接收方上一个正确收到的包是 Packeti-1”，两个事实等价。因此 Rdt2.2 使用 ACKsequence 代替 ACK/NAK ; 从接收方的视角看，“收到损坏的包”和“收到序列号不匹配的包”两种情况行为相同，因此 Rdt2.2 合并了这两条状态转移。


    (5). Rdt3.0

     假设：信道既可能发生错误，也可能丢失分组。
     发送方发送的分组、接收方返回的 ACK 都可能会发生丢失，解决的方法是让发送方等待一个合理的时间，这个时间过后，如果没收到 ACK，则重传数据包。数据包可能由于延迟而并未真正丢失，但由于重传而造成的重复分组问题已经通过序列号机制解决了。


     在 Rdt3.0 中，发送方重传的原因总结如下：

    • ① 传输的分组发生位错误；
    • ② 接收方返回的 ACK 损坏；
    • ③ 数据包丢失，超时；k
    • ④ 数据包未丢失，但定时器设置过短。
     从 Rdt3.0 的 FSM 中可以看到，发送方由于其他原因需要执行的重传，都通过超时需要的重传执行。其意义在于：如果发送方直接重传，虽然 Rdt3.0 也可以正常工作，但如果定时器设置过短，由 ①② 引起的重传随后会导致大量不必要的超时重传。

     定义信道的利用率为：传输数据用的时间与总的发送时间之比。停-等协议信道利用率则为 $$\frac{L/R}{\mathrm{RTT}+L/R}$$ Rdt3.0 可以正确工作，但性能很差。

    2. 滑动窗口协议

    (1). 概述

     针对停-等协议利用率低的原因，利用流水线机制，允许发送方在收到 ACK 之前连续发送多个分组。因此利用率为 $$\frac{n×L/R}{\mathrm{RTT}+L/R}$$ 其中 $n$ 的值取决于窗口大小和 $\small \mathrm{RTT}$ 的大小。
     下面介绍两种滑动窗口协议 (sliding-window protocol)。

    (2). GBN 协议 (Go-Back-N)

     图示为发送方视角的序号范围。基序号 $\mathsf{base}$ 为最小的已发送但未确认的序号，$\mathsf{next\_seq}$ 为最小的未使用序号。



    发送方的窗口： $0,\ 1,\ ...\ , \ \boxed{\mathsf{base},\ ...\ , \mathsf{next\_seq},\ ..., \mathsf{base+\small N-1}}, \mathsf{base+ \small N}$

    • 发送方响应事件：
      • ① 上层调用：发送方首先检查发送窗口是否已满，如果未满则生成分组并发送、并更新变量；否则，把数据返回给上层。（在实际实现时，使用缓存或同步机制)
      • ② 超时：如果发生超时，发送方重传所有已发送但还未确认的分组；发送方仅使用一个定时器，它被用给最早发送但还未被确认的分组。在收到一个 ACK 后，如果仍有未被确认的分组，重启定时器；否则停止定时器。
      • ③ 收到一个 $\small \mathsf{ACK_i}$：GBN 采用累积确认的机制。发送方更新 $\mathsf{base}$ 和定时器；对接收方来说，如果一个 $\small\mathsf{Packet_n}$ 被收到且按序，则接收方发送 $\small\mathsf{ACK_n}$，表明它收到了 n 之前且包括 n 的所有分组，并向上层交付数据；在发送方的视角，可能没有收到某个 $\small \mathsf{ACK_{n-i}}$，但只要收到 $\small\mathsf{ACK_n}$，$\mathsf{base}$ 就移动到 n + 1 的位置。

    接收方的窗口：$... \, \boxed{\mathsf{expected}}, ...$
    * $\mathsf{base \leq expected \leq base+\small N}$。
    * 假设接收方可能收到 $ \small \mathsf{Packet}_i$，则 $\small\mathsf{expected-\small N\leq i \leq base+\small N-1}$，从发送方的视角看则是 $\small \mathsf{base-\small N\leq i \leq base+\small N-1}$，从接收方的视角看则是 $\small \mathsf{expected-\small N\leq i \leq expected+\small N-1}$。

    • 接收方响应事件：
      • ① 收到按序的 $\small \mathsf{Packet_{expected}}$：返回 $\mathsf{\small ACK_{expected}}$、向上层交付数据，然后向前移动一格滑动窗口；
      • ② 收到 $\small \mathsf{Packet_{i}}$，其中 且 $\mathsf{i \ne expected}$。此时接收方丢弃该分组，并返回 $\small \mathsf{ACK_j}$，$\mathsf{j}=$ 最近收到的正序分组编号。

    (3) SR 协议 (Selective Repeat)

     GBN 协议存在性能问题：当窗口长度和时延带宽积很大，流水线中有很多分组时，单个分组的差错就会引起大量不必要的重传。

    发送方的窗口： $0,\ 1,\ ...\ , \ \boxed{\mathsf{send\_base},\ ...\ , \mathsf{next\_seq},\ ..., \mathsf{send\_base+\small N-1}}, \mathsf{send\_base+ \small N}$

    • 发送方响应事件：
      • ① 上层调用：同 GBN。
      • ② 超时：每个分组都有自己的逻辑定时器。
      • ③ 收到一个 $\mathsf{\small ACK_{i}}$：如果 $\mathsf{i}$ 在窗口内，则标记为已接收；如果 $\mathsf{i=send\_base}$，窗口基序号移动到最小已发送未确认序号处；如果窗口发生移动，且有可发送序号，则发送这些分组。

    接收方的窗口： $0,\ 1,\ ...\ , \ \boxed{\mathsf{rcv\_base},\ ..., \mathsf{rcv\_base+\small N-1}}, \mathsf{rcv\_base+ \small N}$
    * $\mathsf{send\_base \leq rcv\_base \leq send\_base+\small N}$。
    * 假设接收方可能收到 $ \small \mathsf{Packet}_i$，则 $\small \mathsf{rcv\_base-\small N\leq i \leq send\_base+\small N-1}$，从发送方的视角看则是 $\small \mathsf{send\_base-\small N\leq i \leq send\_base+\small N-1}$，从接收方的视角看则是 $\small\mathsf{rcv\_base-\small N\leq i \leq rcv\_base+\small N-1}$。

    • 接收方响应事件：
      • ① 在 $\small\mathsf{[rcv\underline{~~} base,rcv\underline{~~} base+\small N-1]} $ 内的分组被接收：即收到的分组落在接收方的窗口范围内。首先返回一个 ACK；如果该分组以前未收到，缓存该分组；如果该分组等于基序号 $\mathsf{rcv\underline{~~}base}$，则将连续的缓存分组交付上层，接收窗口移动。
      • ② 在 $\small\mathsf{[rcv\underline{~~} base-\small N,rcv\underline{~~} base-1]} $ 内的分组被接收：即接收方已确认接收并在窗口外的重复分组，返回一个 ACK。
      • ③ 其他情况，忽略收到的分组。

    (4) 连续 ARQ 协议的窗口大小

    假设有 $n$ bit 可以用于存储序列号，发送窗口大小为 $N_s$，接收窗口大小为 $N_r$。

    ① GBN 协议的发送窗口最大值为 $2^{n}-1$
    [证] 首先，发送窗口内不能有重复编号的帧，有 $N_s\leq 2^n$。
    以 $n=3$ 为例，假设发送方未收到任何确认，接收方收到了全部的帧，此时，
    发送窗口为: $\fcolorbox{blue}{white}{0\ 1\ 2\ 3\ 4\ 5\ 6\ 7}\ 0\ 1\ 2\ 3\ ...$
    接受窗口为: $0\ 1\ 2\ 3\ 4\ 5\ 6\ 7\ \fcolorbox{blue}{white}{0}\ 1\ 2\ 3\ ...$
    接收方无法判断帧 $0$ 是新包还是重传包。由此可得，发送窗口的旧范围不能与接收窗口的新范围有重叠，即 $N_s+N_r\leq 2^n$，当 $N_r=1$，$N_s\leq 2^n-1$。

    ② SR 协议的发送窗口和接收窗口之和的最大值为 $2^n$，且它们各自的最大值为 $2^{n-1}$
    [证] 由 ① 已知，$N_s+N_r\leq 2^n$。且必须有 $N_r\leq N_s$，以保证接收窗口不会溢出。如果 $N_r>N_s$，多出来的部分并没有意义。因此，$N_r=N_s$。
