    3.Electronic Email

    1. E-mail 应用的构成

     同普通邮件一样，电子邮件也是异步通信媒介，人们只在方便的时候收发。
     如图所示，因特网的电子邮件系统主要有三个组成部分：

    • 用户代理 (user agent)
    • 邮件服务器 (mail server)
    • 简单邮件传输协议 (Simple Mail Transfer Protocol, SMTP)
    当用户完成邮件的编辑时，用户代理向服务器发送邮件；当用户想要接受邮件时，用户代理从服务器中接收邮件。
     每个用户在某个邮件服务器上有一个邮箱 (mailbox)，服务器根据邮箱管理维护发给相应用户的报文；邮件服务器之间根据 SMTP 协议通信。E-mail 通信的常见过程为：
    • Alice 为用户代理提供 Bob 的邮箱地址，撰写消息、通过代理发送报文到 Server1；
    • Server1 把报文放入消息队列，运行在 Server1 上的 SMTP 客户端发现这个报文后，创建一个到 Server2 的 TCP 连接；
    • 经过 SMTP 握手后，Server1 发送 Alice 的报文；
    • Server2 收到报文后，把邮件放入 Bob 的邮箱中；
    • 在 Bob 方便的时候，他通过用户代理，读取报文。
    如果 Server2 故障或关机，Alice 的 Server1 会把报文放在消息队列中，一段时间后重新发送；如果一定时间后仍然失败，Server1 以邮件的形式通知 Alice 错误信息；注意到，SMTP 一般不使用中间服务器转发邮件。

    2. SMTP

     SMTP 由 RFC 5321 定义。邮件服务需要可靠性，因此，SMTP 使用持续性 TCP (Port: $25$)，并且 SMTP 也会进行应用层的握手。
     初始的 SMTP 协议可以追溯到 1982 年，它限制邮件的消息体只能通过 $7$ 位 ASCII 码传输。在 20 世纪 80 年代早期，这种限制是明智的，因为当时不会通过邮件传送附件。而现在，通过 SMTP 传送邮件之前，需要将多媒体的二进制数据转化为 $7$ 位 ASCII 码，并且传输过后还需要还原。

    (1). SMTP 通信过程

     首先，SMTP 客户端通过端口号 $25$ 向 SMTP 服务器发起连接，如果服务器没有开机，客户端会在稍后再次尝试。连接建立以后，C-S 之间执行应用层的 SMTP 握手，握手过程中，SMTP 客户端会指示两个用户的邮箱地址。
     假设 SMTP 客户端的主机名为 crepes.fr，服务器的主机名为 hamburger.edu，在建立 TCP 连接之后，SMTP 交互过程如下：

     S: 220 hamburger.edu
 C: HELO crepes.fr
 S: 250 Hello crepes.fr, pleased to meet you
 C: MAIL FROM: <alice@crepes.fr>
 S: 250 alice@crepes.fr ... Sender ok
 C: RCPT TO: <bob@hamburger.edu>
 S: 250 bob@hamburger.edu ... Recipient ok
 C: DATA
 S: 354 Enter mail, end with ”.” on a line by itself
 C: Do you like ketchup?
 C: How about pickles?
 C: .
 S: 250 Message accepted for delivery
 C: QUIT
 S: 221 hamburger.edu closing connection
     在这个例子中，邮件服务器 crepes.fr 向邮件服务器 hamburger.edu 发送了一段报文“Do you like ketchup? How about pickles?”。SMTP 客户端在整个过程中发送了五个命令：HELO、MAIL FROM、RCPT TO、DATA、QUIT，服务器对这些命令作出回答，包含回答码和可选的解释。客户端发送一个只有“.”的行 (对应于 ASCII 码的 CLRF.CLRF)，指示断开连接。

    (2). 与 HTTP 的对比

    • HTTP 是拉式协议，用户从服务器拉取数据；SMTP 是推式协议，发起 TCP 连接的 SMTP 服务器把文件推向接收服务器。
    • HTTP 的二进制数据无需进行编码解码。
    • HTTP 把对象封装到自己的相应报文中，SMTP 把所有的东西封装到一个报文中。

    3. 邮件报文格式

     邮件报文的格式由 RFC 5322 定义。类似于 HTTP，头部行和消息体之间用 CRLF 隔开，且一部分首部行是必要的，包括 From、To、Subject。
     例子： 

    1
2
3
4
5
    From: alice@crepes.fr
To: bob@hamburger.edu
Subject: Searching for the meaning of life.

// content
     多用途网际邮件扩展 (Multipurpose Internet Mail Extensions, MIME) 广泛应用于电子邮件系统和浏览器，它定义于 RFC 2045-2049。服务器可以通过增加额外的头部行标识发送数据的类型，改进了传统的基于 SMTP 的邮件格式的限制。

    4. 邮件访问协议

     20 世纪 90 年代以前，用户通过登录，直接在服务器主机上运行一个阅读程序读取邮件。今天的邮件访问使用了 C/S 结构，即用户在自己的主机上阅读邮件，主机上运行用户代理。
     回顾 Alice 向 Bob 发送邮件的完整过程。Alice 的用户代理向 Server1 发送报文的过程也是把报文推向一个目的主机，因此可以通过 SMTP 完成；而 Bob 通过代理读取邮件是一个拉操作，需要引入邮件访问协议。

    (1). POP3

     POP3 (Post Office Protocol) 协议是一个极为简单的邮件访问协议。POP3 使用 TCP，且在邮件服务器的 $110$ 端口工作。在 POP3 的整个过程中，服务器对用户的命令作出应答，可能的应答有两种：

    • +OK: 用于指示前面的命令是正常的；
    • -ERR: 用于指示前面的某个命令出了差错。
    POP3 服务器的工作流程可以分为三个阶段：
    • 认证 (authorization): 用户代理以明文的方式发送用户名和口令，以鉴别用户。涉及两个命令：user <username>、pass <password>。
    • 事务处理：获得报文，同时可以控制报文的删除标记。
      • “下载并删除”：用户代理先后使用 list (列出所有报文的长度)、retr (取回邮件)、dele (标记邮件为删除) 命令。
      • “下载并保留”：在服务器端保留邮件。
    • 更新：在 quit 命令后，会话结束，服务器进入更新阶段，删除被标记的邮件。
     POP3 服务器虽然保留了一些状态信息，但会话过程并不会携带状态信息。因此，POP3 是无状态协议。

    (2). IMAP

     IMAP (Internet/Interactive Mail Access Protocol) 在 POP3 的基础上，支持用户远程创建文件夹、在文件夹之间移动邮件或查询邮件。IMAP 服务器维护用户会话的状态，例如文件夹的名字等。
     IMAP 另一个重要特性是它支持用户代理只读取报文的某个部分。当用户代理与服务器以低带宽连接时，这个特性非常有用。

    (3). HTTP

     今天，绝大多数的邮件服务都提供基于 Web 的接入。在这种情况下，用户代理就是浏览器：发送代理通过 HTTP 发送邮件，接受代理也通过 HTTP 访问邮件，但邮件服务器之间仍使用 SMTP。
