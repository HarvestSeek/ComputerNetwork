    2.Web And HTTP

    1. Web 概述

     20 世纪 90 年代以前，因特网的主要使用者是研究人员、学者和大学生。20 世纪 90 年代初期，Tim Berners-Lee 发明的万维网 (World Wide Web) 登上历史舞台，极大地提升了互联网的地位。
     Web 使用的应用层协议是超文本传输协议 (HyperText Transfer Protocol, HTTP)。浏览器通过读取服务器发来的 HTML (HyperText Markup Language) 文件浏览网页。

    • 对象：HTML 文件、图片、视频、动态脚本等。
    • 网页 (Web Page): 由多个对象组成。一个网页通常包含一个基本的 HTML 文件，它包含引用其他对象的链接。
    • URL: Uniform Resource Locator，统一资源定位器，用于寻址、定位文件，格式为 (Scheme)://host:port/path。

    2. HTTP

    (1). 基本概念

     HTTP 定义了 Web 客户端向服务器请求网页以及 Web 服务器传送网页的方式。HTTP 使用 TCP 作为传输层协议，C-S 之间建立连接。
     HTTP 是一个无状态的协议 (stateless): 服务器不会维护过去的有关客户端所发请求的消息，即对事务处理没有记忆能力，只是机械地根据收到的报文做出回应，不需要状态转换 1。因此，Web 服务器是始终打开的。

    (2). 连接类型

    ①. 非持久性连接

     使用非持久性连接的 HTTP 应用，每一对 request/response 各自经一个 TCP 连接发送。定义响应时间 $=$ 客户请求 HTML 基本文件到客户收到整个 HTML 文件所花的时间；衡量它的单位是往返时间 (Round-Trip-Time, RTT)，指客户端发送一个很小的数据包到服务器并返回所需要的时间。
     由于 TCP 连接在传输数据前需要握手，因此，每个对象的响应时间 $=2×\mathrm{RTT}\ +$ 文件发送时间。

    ②. 持久性连接

     每个 TCP 连接允许传输多个对象，在建立连接后，服务器会保持连接的打开状态。

    • 无流水线的持久性连接：客户端只有在收到一个 response 后才会发送新的 request。每个被引用的对象耗时一个 $\mathrm{RTT}$。
    • 带有流水线的持久性连接：HTTP 1.1 的默认选项，理论上收到所有被引用的对象只需要一个 $\mathrm{RTT}$。

    e.g. 传输一个引用 $5$ 个 JPEG 图像的 HTML 网页，HTTP 1.0 需要 $12$ 个 $\mathrm{RTT}$，HTTP 1.1 需要 $7$ 个 $\mathrm{RTT}$，带流水的 HTTP 1.1 需要 $3$ 个 $\mathrm{RTT}$。

    • 请求行 (request line): 格式为 |method|sp|URL|sp|version|cr|lf|。
      • GET: 请求服务器的指定资源。使用此方法时消息体字段为空。对比 POST，如果输入的数据不长，可以省出消息体字段，在 URL 中填入输入的信息，例如 www.somesite.com/animalsearch?monkey&banana。

    • 头部行/请求头 (head lines): 格式为 |header field name:value|cr|lf|... ...。
      • Host: 请求的主机名
      • User-Agent: 产生请求的浏览器名
      • Connection: 当 HTTP 1.1 需要非持久连接时使用这个头部行，并设其值为 close。

    3. Cookie

     HTTP 是一个无状态的协议，这简化了 Web 服务器的设计。但 C-S 交互过程中，服务器通常希望可以识别用户，因此 HTTP 使用了 Cookie。

    (1). Cookie 组件

    ① HTTP 响应报文中的一个 Cookie 头部行；
    ② HTTP 请求报文中的一个 Cookie 头部行；
    ③ 用户的主机中保存的一个 Cookie 文件，由浏览器管理；
    ④ 位于 Web 服务器端的一个后台数据库。

    (2). Cookie 原理

     如图，假设用户过去访问过 eBay，且首次访问 amazon.com。
     当请求到达服务器后，Web 服务器产生一个唯一的识别码，用以在后端数据库中索引一个表项。然后 Web 服务器在响应报文的头部行中添加一条 Set-Cookie: 1678。
     随后，如果客户继续访问 Amazon，浏览器就会在 Cookie 文件中抽取识别码，并添加到请求报文的首部行中：Cookie: 1678。实际上的 Cookie 需要经过加密。

    4. Web 缓存

     Web 缓存即代理服务器，目的为减少请求时间，提升性能，减少流量开销，实现有效内容分发。

    (1). 缓存应用实例

     假设机构网络是一个高速局域网，网关与 Internet 通过一条 $15\mathit{Mbps}$ 的链路连接；假设对象平均大小 $1 \mathit{Mb}$，机构内的浏览器对服务器的平均访问速率为每秒 $15$ 个请求；假设因特网时延为 $2\ \mathrm{s}$，即网关发送请求和收到相应的时间间隔。
     接入链路上的流量强度为：$15×1\mathnormal{Mb} /15\mathnormal{Mbps}=1$，时延很大且无限增长；局域网内 $0.15$ 的流量强度可以被忽略。
     可以通过提升接入链路的带宽解决问题，比如增加到 $100\mathit{Mbps}$，但成本高昂。更好的办法是在机构网络中增加一个代理服务器，实践中的命中率通常介于 $0.2\sim 0.7$。假设命中率为 $0.4$，此时接入链路的流量强度为 $0.6$，时延约为几十毫秒，与因特网时延的 $2$ 秒相比可以忽略。局域网内的时延为 $10$ 毫秒，因此，总的平均时延为 $0.4×0.01+0.6×2.01\approx 1.2\ \mathrm{s}$。这种解决方案的成本很低。

    (2). 条件 GET 方法

     代理服务器在返回对象时，有时需要考虑返回的对象是否为最新版本，可以通过 HTTP 的条件 GET 方法实现。
     代理服务器向服务器第一次请求某个资源时，服务器返回一个携带资源的 200 OK 报文，其中一个头部行为 Last-Modified: <date>。一定时间过后，如果这个对象被某个主机请求，代理可能会确定它是否为最新版本，它会向服务器发送一个 GET 请求，其中一个头部行包含 If-Modified-Since: <date>。如果没有被修改过，服务器会返回一个 304 Not Modified 报文；否则返回一个带有数据的 200 OK 报文。
