    4.DNS

    1. DNS 服务

    (1). DNS 概述

     DNS (Domain Name System, 域名系统) 是一个由 DNS 服务器实现的分布式数据库，也是主机查询这个分布式数据库使用的应用层协议，运行于 UDP 之上、使用 $53$ 端口。
     一台主机通过 Web 中输入的域名访问服务器的时候，还需要获得服务器的 IP:

    • 浏览器从 URL 中抽取出域名，并传给主机上的 DNS 客户端
    • DNS 客户端向服务器发起一个包含主机名的请求，收到应答后，提取出其中的 IP
    这个额外的过程造成一定的时延，且有时候很客观。但通常情况下，目的 IP 会缓存在附近的某个 DNS 服务器中。

    (2). DNS 提供的服务

    • 域名向 IP 的转换。
    • 主机别名：通过主机的别名获取规范主机名。
    • 负载均衡：对于繁忙的服务器，它的域名对应于多台服务器主机，DNS 服务器维护这个域名到 IP 地址集合的映射；当用户请求目的 IP 时，DNS 服务器在这些 IP 之间循环分配负载。

    2. DNS 工作原理

    (1). 分布式数据库

     如果 DNS 服务器被设计成集中式的单一服务器，会造成诸如单点故障、通信容量、时延、难以维护等问题，因此 DNS 采用了分布式、层次化的设计。

    • 根域名服务器：有 $400$ 多个根域名服务器遍及世界，保存到 TLD 服务器的地址。
    • 顶级域名服务器 (Top-Level Domain, TLD)：每个顶级域名 (如 com、org、net、edu) 和国家顶级域名 (如 fr、cn) 都有其对应的 TLD 服务器，保存了到权威域名服务器的地址。
    • 权威域名服务器：每个组织在其中提供公共可访问的 DNS 记录映射，提供组织内部服务器解析。
     本地域名服务器不属于层次结构，但也是 DNS 系统中关键的一部分。每个 ISP 会为接入的主机提供一个或多个本地域名服务器的 IP 地址 (通常通过 DHCP)，它与主机邻近，起到代理的作用。

    (2). DNS 查询

     假设主机 cse.nyu.edu 想知道主机 gaia.cs.umass.edu 的 IP，同时假设 cse.nyu.edu 的本地 DNS 服务器为 dns.nyu.edu。

     cse.nyu.edu 首先向本地域名服务器发送一个 DNS 查询报文，本地域名服务器将此报文转发至权威域名服务器，然后权威域名服务器根据 edu 前缀返回相应的 TLD 服务器的 IP 列表，本地域名服务从中选择一个，以此类推。在这个例子中，共发送了 $8$ 份 DNS 报文。实际上，TLD 服务器可能无法一次定位到目的权威域名服务器，在机构内部的权威域名服务器可能也存在层次关系，这时就需要更多的查询次数。

     上面的例子中，cse.nyu.edu 向本地域名服务器的查询是递归查询，把域名解析的任务交给了所查询的服务器；而本地域名服务器的查询是迭代查询，被查询的服务器直接返回域名解析服务器的地址。理论上，任何 DNS 都可以是递归或迭代的，但一般的查询方法都采用上面的模式。
     右图为所有查询都是递归的一条查询链。

    (3). DNS 缓存

     (2) 中的讨论忽略了 DNS 缓存，实际的 DNS 广泛应用了缓存技术。在一个请求链中，当某个 DNS 服务器收到 DNS 应答报文，获得主机名 → IP 的映射，就会缓存这个映射。由于主机名到 IP 的映射不是永久的，DNS 服务器通常在两天后丢弃缓存信息。
     在 (2) 的第一种通用的查询方式中，本地域名服务器会缓存 TLD 服务器的 IP，因此根域名服务器很少会被访问。

    3. DNS 记录和报文

    (1). DNS 资源记录

     实现 DNS 分布式数据库的所有域名服务器都存储了资源记录 (Resource Record, RR)，资源记录是一个包含下列字段的四元组：(Name, Value, Type, TTL)。

    • Type = A，则 Name 是主机名，Value 是它对应的 IP。
    • Type = NS，则 Name 是个域 (如 foo.com)，Value 是这个域对应的权威域名服务器的主机名。
    • Type = CName，则 Name 是主机别名，Value 是它对应的规范主机名。
    • Type = MX，则 Name 是主机别名，Value 是它对应的邮件服务器的规范主机名。MX 记录允许邮件服务器有简单的别名。DNS 客户端通过 MX 记录获取获取邮件服务器的规范主机名，通过 CName 记录获取其他服务器的规范主机名。

    (2). 举例：注册一个域名

     假设我们创建了一个公司 Network Utophia。我们通过某些机构注册域名 networkutophia.com，需要提供基本的和辅助的权威域名服务器的主机名和 IP。以基本权威域名服务器为例，假设相关信息为 dns1.networkutophia.com、212.212.212.1。
     注册机构在 TLD com 服务器插入一条 NS 记录和一条 A 记录：
      (networkutophia.com, dns1.networkutophia.com, NS)
      (dns1.networkutophia.com, 212.212.212.1, A)
     最后，我们要保证用于 Web 服务器 www.networkutophia.com 的 A 记录和用于邮件服务器 mail.networkutophia.com 的 MX 记录插入到我们的权威域名服务器中。这样，外界就可以访问我们的 Web 站点，并向我们发送邮件。
     当外界要访问我们的 Web 站点，在查询过程中，TLD 服务器会向外界的本地域名服务器返回这两条 A 记录和 NS 记录，随后向 212.212.212.1 查询 Web 服务器对应的 A 记录。

    (3). DNS 报文

     DNS 只有查询和回答两种报文，并且这两种报文格式相同。

    • 前 $12$ 个字节是首部区域。
      • Identification: 用于标识查询报文，这个编号会被写回到回答报文中。
      • Flags: 包括查询/应答标志、权威标志 (如果接收服务器是目的权威域名服务器)、希望递归标志等。
      • 数量字段：指出后面四类数据区域出现的数量。

    • 问题字段：包含当前查询的信息。
    • 回答字段：包含对应查询的 RR 记录。
    • 权威字段：其他权威服务器的记录。
    • 附加字段：其他有帮助的记录。
